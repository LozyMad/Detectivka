# Диагностика: почему сайт перестаёт работать

## Что добавлено в проект

1. **Глобальные обработчики ошибок** (`backend/server.js`)
   - `uncaughtException` — любая необработанная ошибка логируется с полным стеком, затем процесс завершается.
   - `unhandledRejection` — необработанный промис (например, забытый `catch` в async-функции) логируется, затем процесс завершается.

2. **Логирование каждого запроса**  
   В консоль пишется строка вида:  
   `2025-03-12T10:00:00.000Z GET / 200 5ms`  
   По ним видно, доходят ли запросы до сервера и не «зависает» ли он.

3. **Обработка ошибок Express**  
   Ошибки из маршрутов перехватываются централизованно, в лог пишется `[Express error]` и стек.

4. **PostgreSQL**
   - Ошибки пула соединений (обрыв связи с БД) логируются как `[PostgreSQL pool error]`, процесс не падает из-за них.
   - Заданы таймауты: `connectionTimeoutMillis: 10000`, `idleTimeoutMillis: 30000`, чтобы «зависшие» соединения не копились.

## Как искать причину

### Запуск с сохранением логов в файл

**Windows (PowerShell):**
```powershell
cd c:\Users\lozym\Desktop\Detectivka
node backend/server.js 2>&1 | Tee-Object -FilePath server.log
```

**Или просто перенаправить вывод:**
```powershell
node backend/server.js > server.log 2>&1
```

Держи консоль открытой и пользуйся сайтом. Когда сайт «перестанет работать», открой `server.log` и посмотри последние строки:

- **`[CRASH] uncaughtException`** или **`[CRASH] unhandledRejection`** — процесс упал из-за ошибки; по стеку будет видно файл и строку.
- **`[PostgreSQL pool error]`** — проблемы с базой (сеть, недоступность БД).
- **`[Express error]`** — ошибка в одном из API-маршрутов.
- Если в конце лога только запросы (GET/POST с 200), а потом пусто — возможен обрыв соединения с хостингом или завершение процесса извне (например, при закрытии консоли).

### На Timeweb (или другом хостинге)

В панели хостинга открой **логи приложения** (или stdout/stderr контейнера/сервиса). Там будут те же сообщения: `[CRASH]`, `[PostgreSQL pool error]`, `[Express error]` и строки с запросами.

## Частые причины «перестаёт работать»

| Симптом в логах | Возможная причина |
|-----------------|-------------------|
| `unhandledRejection` / `uncaughtException` | Ошибка в коде или в цепочке промисов без `catch`. |
| `PostgreSQL pool error` / таймауты к БД | БД недоступна, обрыв сети, исчерпан лимит соединений. |
| Логи обрываются, новых запросов нет | Закрыта консоль/сессия (при запуске «через консоль») или хостинг завершил процесс. |
| Много запросов, потом тишина | Возможна утечка памяти или «зависание» event loop (долгая синхронная работа). |

## Рекомендация для продакшена

Не запускай приложение «просто в консоли» на сервере: при закрытии сессии процесс завершится.

- **На VPS/сервере:** используй менеджер процессов, например **PM2**:
  ```bash
  npm install -g pm2
  pm2 start backend/server.js --name detectivka
  pm2 logs detectivka   # просмотр логов
  pm2 restart detectivka
  ```
- **На Timeweb:** если есть «автодеплой из Git» и свой runtime, уточни в документации, как у них принято держать Node-процесс запущенным (systemd, PM2, Docker и т.п.) и где смотреть логи.

После следующих падений смотри логи по этой инструкции — по ним можно будет точно указать причину и исправить код или настройки БД/хостинга.
